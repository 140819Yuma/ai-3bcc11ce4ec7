# AIリサーチ スキル

## スキルの概要

このスキルは、AIエージェント開発に関するリサーチを実施し、その結果を `index.html` に蓄積していくためのワークフローを定義する。
Yumaとのプロジェクトで「調べたこと・気づいたこと・設計決定」を体系的にまとめるための共通手順。

---

## 発動タイミング

以下のような依頼があった場合にこのスキルを使用する：

- 「〇〇について調べて、HTMLに追加して」
- 「リサーチしてまとめて」
- 「このPDF/動画の内容をHTMLにまとめて」
- 「〇〇のセクションをindex.htmlに追加して」
- 「AIリサーチを実行して」

---

## 実行手順

### STEP 1: インプットを把握する

まず何を調査するかを明確にする。ソースは以下のいずれか：

1. **ユーザーのメッセージ**（「〇〇について調べて」）
2. **PDFファイル**（`Read` ツールでフォルダ内のPDFを読む）
3. **URL・YouTube動画**（`WebFetch` でタイトル・説明取得、`WebSearch` で補完）
4. **既存の `index.html`**（現在の内容を把握してから更新）

#### 受付チェック（入力不足で止まらないために）

以下が不明な場合は、**1回だけまとめて質問**し、回答がなければカッコ内のデフォルトで進める：

- 調査の深さ：概要レベルか実装レベルか（→ デフォルト: 概要＋主要な実装ポイント）
- 追記先：既存セクションか新セクションか（→ デフォルト: 内容で判断し、判断理由を報告）
- 目的：自分たちのプロトタイプに使うのか、Kokiへの共有用か（→ デフォルト: 両方を意識）

### STEP 2: 現在の index.html を確認する

```bash
# 現在のHTMLの構造を把握
Read /Users/kaminagayuma/Library/CloudStorage/OneDrive-u-aizu.ac.jp/デスクトップ/AI作業/index.html
```

- 既存のセクション一覧を把握
- 新しいコンテンツをどこに追加するか決定
- 重複がないか確認

### STEP 3: リサーチを実施する

調査方法は並列で行う：

```
WebSearch（日本語・英語） + WebFetch（詳細ページ）を同時実行
```

調査対象の例：
- アーキテクチャパターン・設計手法
- フレームワーク比較（LangGraph / CrewAI / AutoGen 等）
- API仕様・料金・制限値
- 実装コードサンプル
- ベストプラクティス・アンチパターン

### STEP 4: index.html に追記・更新する

#### 追記ルール

1. **既存セクションへの追記**：関連するセクションに情報を加える
2. **新しいセクション追加**：テーマが独立している場合は `<section id="新id">` を追加
3. **進捗ログの更新**：`#log` セクションに日付付きのエントリを追加

#### HTMLの統一スタイルルール

- ダークテーマ（`--bg: #0d1117` ベース）を維持
- セクション構造：`.sec-tag` → `.sec-title` → `.sec-sub` → コンテンツ
- カードグリッド：`.card-grid` > `.card`
- コードブロック：`.code-wrap` > `.code-head` + `<pre>`
- テーブル：`.tbl-wrap` > `<table>`
- 注意書き：`.callout.c-info/.c-warn/.c-tip/.c-danger`
- ナビゲーションの `<ul>` にアンカーリンクを追加

#### 追記の際に含めるべき情報

| 種類 | 含める内容 |
|------|-----------|
| 技術調査 | 仕組み・メリット・デメリット・実装コード例 |
| 比較 | テーブル形式で並列比較 |
| 進捗ログ | 日付・完了事項・気づき・次のアクション |
| API情報 | 無料枠の制限値・推奨モデル・コスト感 |
| 設計決定 | なぜその選択をしたか・トレードオフ |

### STEP 5: QAゲート（追記後に必ず実行）

以下をすべて確認してから完了報告する：

- [ ] HTMLの構文が壊れていない（`python3 -c "from html.parser import HTMLParser; HTMLParser().feed(open('index.html', encoding='utf-8').read())"` がエラーなく通る）
- [ ] ナビゲーションの `<ul>` に新セクションへのアンカーリンクを追加した
- [ ] 追記した情報すべてに**出典URL＋取得日**を付けた
- [ ] `#log` セクションに日付付きエントリを追加した（何を調べ、何がわかり、次に何をするか）
- [ ] 既存セクションと内容が重複していない
- [ ] 推測と事実を区別した（未確認情報は「未確認」と明記）
- [ ] APIキー等の機密情報を書いていない

完了報告には「追加したセクション名」「主な発見3点以内」「次のアクション候補」を含める。

### ディスカッションサイクルモード

「ディスカッションサイクルを回して」「HTMLを改善して」「調査と議論して」と言われた場合の特別モード。
**目的: 調査⇔既存HTMLの突き合わせ議論でレポートの質を上げ続け、Yumaは意思決定ボードのチェックだけで開発方針を決められる状態にする。**

1サイクル = 以下の5フェーズ：

#### Phase A: 棚卸し
`index.html` の主要な主張・数値・設計判断を箇条書きで抽出する（特に `#plan` `#hybrid` `#free-api` `#pricing` `#log`）。それぞれに「いつ時点の情報か」を付ける。

#### Phase B: 調査
WebSearch（日英並行）で最新動向を調べる。定点観測テーマ：
- マルチエージェント設計論の変化（single vs multi、オーケストレーションパターン）
- フレームワーク動向（LangGraph / CrewAI / OpenAI Agents SDK / Claude Agent SDK 等）
- Groq / Gemini の無料枠・レート制限・モデルの変更
- 業務導入・観察日誌系の面白い事例
- 加えて、前回サイクルの意思決定ボードで「深掘り」指定されたテーマ

#### Phase C: ディスカッション（このモードの核心）
調査結果と既存HTMLの主張を突き合わせ、以下の4分類で議論する：

| 分類 | 意味 | HTMLへの反映 |
|------|------|------------|
| 🟢 裏付け | 既存の設計判断が最新動向とも整合 | 該当箇所に根拠を追記 |
| 🟡 更新 | 数値・状況が古くなった | 該当箇所を更新（旧情報は取り消し線 or 注記で残す） |
| 🔴 矛盾 | 既存の主張と最新動向が食い違う | 両論併記し、意思決定ボードに論点として起票 |
| ⚪ 欠落 | HTMLに視点がない | 新規追記 or 意思決定ボードで要否を確認 |

**重要：勝手に設計判断を書き換えない。** 🔴矛盾は必ず両論併記＋意思決定ボード行きにする。

#### Phase D: 反映
1. 🟢🟡⚪ をHTMLに反映（出典URL＋取得日つき）
2. `#decisions`（意思決定ボード）に🔴と⚪の要判断項目を起票する。1論点 = 「背景 / 選択肢（2〜3個） / AIの推奨と理由 / ステータス：未決」の形式
3. Yumaが決定したら、その論点のステータスを「✅ 決定：〜」に更新し、決定内容を本文に反映する（次サイクルの冒頭で確認）

#### Phase E: QAゲート
下記STEP 5のQAゲートを通し、完了報告には「議論の結果（🟢🟡🔴⚪の件数と要点）」「意思決定ボードの未決論点」を含める。

### 週次ダイジェストモード

「週次ダイジェスト」「今週のAIエージェント動向」と言われた場合の特別モード：

1. 直近7日のニュースを WebSearch で調査（日英並行）：
   - マルチエージェントフレームワークの更新（LangGraph / CrewAI / AutoGen 等）
   - Groq / Gemini の無料枠・モデル・料金の変更
   - 業務導入事例・新しい設計パターン
2. **自分たちのプロジェクトに関係あるものだけ**に絞る（最大5トピック）
3. 各トピック：見出し＋3行要約＋出典＋「うちへの影響」1行
4. `index.html` の `#log` に「YYYY-MM-DD 週次ダイジェスト」として追記
5. STEP 5 のQAゲートを通す

## 注意事項

- PDFやURLを追加リサーチする際は、WebSearch と WebFetch を並列実行して効率化する
- 既存のHTMLスタイル（CSS変数・クラス名）を壊さないように追記する
- 情報の鮮度（取得日）をフッターまたはコメントに記載する
- 機密情報（APIキーなど）はHTMLに直接書かない
- 長くなりすぎる場合は別ファイル（`research/topic-name.html`）に分割することも検討する
