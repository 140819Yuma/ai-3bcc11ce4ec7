Project Overview

プロジェクト概要

業務にAIエージェントを導入することで、AI × 人間のシステムの体系化を進め、人間側のリソースの有効活用をはかる。

🎯

タスク管理の自動化

Google Meetの議事録を自動読み込みし、To Doリストの作成・リマインドを実施。カレンダー連携・要項・報告書の自動作成も含む。

優先度：高
🔍

特定テーマの情報収集を自動化。複数AIが分担して最新ニュース・特定サイトのスクレイピング・資料整理を行う。

課題
🧭

AI導入フレームワーク

「①解決すべき課題の特定 → ②目的の明確化 → ③手法の選定」の順で整理すべきとNobuyuki Ishikawaが指摘。コストを抑えることに固執せず、得られる知見があれば一定の費用を投じて実験する価値がある。

方針
🎯

「業務効率化」を超えた目的設計

単なる「自動化」「効率化」という言葉で終わらせず、具体的なストーリーが必要。Gemini単体で完結する作業ではなく、APIで外部システムと連携して何が実現できるかの仮説設計が重要。

設計思想
🤝

人間とAIの役割分担

本実験のテーマは「人間とAIの役割分担」。予定管理などAIで代替可能なルーチンワークを任せ、人間は人間にしかできない業務に集中できる環境を作ることが目的とKoki Matano・Yuma Kaminagaが共有。

テーマ
📰

ユースケースと話題性

自分たちなりのユースケース・面白いレポートを作ることが重要。既存のAI関連情報を参考にしつつ、話題性のある領域をあえて狙い、自分たちなりの経験を蓄積する戦略。

戦略
🧪

独自性のある実験環境

既存APIを組み合わせ「自分たちがエージェントを活用している」と実感できる環境を作ることが目標。機能実装だけでなく、独自のユニークな体験を得ることを重視すべきとの助言。

設計
⏱️

速度優先の方針

卒業までの限られた時間を考慮し、費用対効果の厳密な計算より「小さな計画で実際に動かしてみる速さ」を優先することで合意。回収モデルの構築より、まず目的地に到達し経験を買う姿勢を重視。

合意事項
🧸

AIエージェントの選択肢が非常に多いため、まずは気軽に楽しみながら学習を進める方針。Yuma・Kokiは今後改めて具体的なツール・手法を調査・検討予定。

実行計画 — 要件を満たす最短ルート

打ち合わせで出た要件を棚卸しすると、7割は既存の資産（プロトタイプ＋本レポート）で満たせる状態にある。埋まっていない最大のギャップは「ストーリーの一本化」。ここでは要件を貫く一本の物語を固定し、無料枠のまま到達できる4段階の計画に落とす。

One Story

「会議が終わったら、人間はもう何もしない。
翌朝、ToDoと日報が届いている」

この一文が打ち合わせの要件をすべて同時に満たす。課題は議事録処理というルーチンワーク（＝役割分担テーマそのもの）。Gemini単体では実現不可能（フォルダ監視・通知というAPI連携が必須）。「朝スマホに届く」という体験の実感。そして既存のPython司令官＋リフレクションループの延長で作れるため、速さ優先の合意にも沿う。

4段階ロードマップ — すべて無料枠のまま

1

今週：フレームワークを1枚に落とす

「課題の特定 → 目的の明確化 → 手法の選定」を One Story で埋めた1枚を作る。課題＝議事録処理・タスク管理のルーチン／目的＝人間は判断だけに集中する環境／手法＝既存Python司令官の拡張。これが次回ミーティングの土台になる。

2

数日：入口と出口をつける

フォルダ監視（inboxに議事録mdを置くと自動処理が走る）＋出力（ToDo.md・日報.mdの生成）。ターミナル対話からの卒業がそのまま「エージェントを使っている実感」に変わる。既存のリフレクションループはそのまま中身として使う。

3

1週間：外部連携を1つだけ足す

最初はDiscordのWebhook通知が最適——無料・審査なし・数行のコードで「朝刊がスマホに届く」体験が完成する。Googleカレンダー連携は認証（OAuth）が重いので2番手に回す。これで「Gemini単体でできないこと」の要件を具体物で証明できる。

4

継続：観察日誌をつける

動かした結果を失敗込みで本レポートに追記していく。面白さ×新規性の調査で確認したとおり、Project VendもAI Villageも価値の中心は性能ではなく観察記録だった。「話題性」「面白いレポート」の要件はこの習慣だけで回収できる。

🎒

次回ミーティングへの持ち物（宿題「各自調査して深掘り」への回答）： ①動くデモ——議事録を放り込むと朝刊が出てくるまでの一連の流れ。②本レポートサイトのURL——調査成果としてそのまま提示できる。③3人で決めたい論点リスト——観察日誌の公開範囲、課金トリガーの承認、次に足す連携先の優先順位。この3点が揃えば担当分としては十分な形になる。

論点 1 ｜ 🔴 矛盾 ｜ ステータス: 未決

クラウド側のリフレクション構成を「司令官＋使い捨てサブエージェント」型に寄せるか

背景： Cognitionは2025年6月の「Don't Build Multi-Agents」で対等な複数エージェント協調（ピア型）は脆いと主張したが、2026年3月に自ら管理者型マルチ（Devinが複数のDevinを管理）へ転換。業界は「単一エージェントが全体の文脈を保持し、使い捨てのサブエージェントが個別タスクを実行して要約だけを返す」形に収束した（本番導入で生き残ったパターン調査、2026-07-02取得）。現行のPython司令官はこの収束形と一致する一方、無料APIセクションの「草案Gemini／品質GPT-4o／修正Groq」というピア型リフレクション構想は脆いとされた形に近い。

選択肢内容備考
A（AI推奨）クラウド側も「司令官がサブエージェントを都度起動→要約回収」に統一業界収束形と一致。Token Gobblerリスク（トークン3〜10倍消費）も抑制
Bピア型リフレクションをあえて維持し、比較を観察日誌のネタにする話題性はあるがトークン消費が重い
C両方作って同一タスクで比較実験面白いが工数増。Step 2〜3完了後なら現実的
論点 2 ｜ ⚪ 欠落 ｜ ステータス: 未決

フレームワーク（LangGraph等）をいつ導入するか

背景： 2026年時点でLangGraphが本番運用の事実上の標準（v0.4でチェックポイント・人間承認フローを強化）、CrewAIは最速プロトタイプ向けという整理が定着（フレームワーク比較、2026-07-02取得）。同一モデルでもフレームワーク選択でベンチマーク最大30ポイント差の報告もある。現行は自作Python司令官で、本レポートにはこの「いつ乗り換えるか」の基準がなかった。

選択肢内容備考
A（AI推奨）Step 2〜3（フォルダ監視＋Discord）までは自作継続。「状態の永続化」か「人間承認フロー」が必要になった時点でLangGraphを検討今の規模なら自作で十分。移行基準が明文化される
B今すぐLangGraphへ移行学習コストでStep 2〜3が遅れる
Cフレームワークなしを貫く規模拡大時に自作で再発明が続くリスク
論点 3 ｜ 🟡 更新 ｜ ステータス: 未決

Groq TPMボトルネック対策 — Gemini再導入のタイミング

背景（数値更新）： Groq無料枠の llama-3.3-70b-versatile は 30 RPM / 1,000 RPD / 12,000 TPM / 100,000 TPD（2026年6月時点、公式ドキュメント）。リフレクションループは1タスクでトークンを3〜10倍消費するため、実質の壁はRPDではなくTPM 12,000。一方Geminiは2025年12月の無料枠削減のあと、2026年5月7日にFlash系の上限が引き上げられ、Flash無料枠は 10 RPM / 250,000 TPM / 1,500 RPD（公式レート制限、2026-07-02取得）。TPMはGroqの20倍以上あり、トークン重量級タスクの逃し先として有力。

選択肢内容備考
A（AI推奨）無課金アカウントでGeminiキーを再作成し、長文タスクをGeminiへ逃す2プロバイダ体制既存の「継続中」タスクと一致。¥0方針を維持
BGroq単体のままプロンプト圧縮・履歴要約で耐える工数小だが根本解決ではない
C少額課金でGroq Devプランへ課金判断基準（無料枠枯渇）にまだ達していない
✅

Organization Design

エージェント設計の3要素 & 24時間運用フロー

組織図そのものはハイブリッド設計セクションの全体構成図と重複するため統合済み。ここでは個々のエージェントを動かす3要素と、CEOが朝起きるまでの24時間自動運転フローを整理する。

個別エージェントの3要素

📖

① マニュアル（指示書）

職務定義のシステムプロンプト＋文脈エンジンから読み込んだ会社のルール。全エージェントは作業前に必ずこれを参照し、ブレのないアウトプットを出す。

例：「あなたはUIデザイナーです。{デザイン言語} に従いコンポーネントを作成してください。」
🧠

② 頭脳（LLM）

役割・難易度でモデルを使い分ける。リーダーAI（Sonnet）が日常タスクを担い、高度な戦略判断が必要な場合のみアドバイザーAI（Opus）にお伺いを立てる。

ルーチン処理 → Sonnet（コスト低）
戦略判断 → Opus（スポット利用のみ）
🗂️

③ 机（独立セッション）

タスクごとに完全分離されたクリーンな実行環境。1つのLLMアカウントを使い回さず、役割ごとに独立したサブエージェントとして動かす。

タスクA・Bは別セッション。終了後は破棄。コンテキスト混入ゼロ。

24時間自動運転フロー

1

CEO がスマホからメモを投入

Obsidianの「インボックス」に思いつきのアイデア・依頼をメモするだけ。

2

タスクディスパッチャー AIが自動スキャン・タスク登録

インフラAIが定期的にインボックスを確認し、タスクボードに自動で「タスク」として登録・チームに割り振り。

3

夜間に専門AIが並列処理・QAがテスト実行

各チームのエージェントがタスクをこなし、QA担当がコードテスト・品質チェックを自動で回す。引き継ぎが必要な場合は「引き継ぎ書（Markdown）」を出力して次のエージェントに渡す。

!

判断に迷ったらエスカレーション（ブロック）

エージェントの権限外の判断が必要になったら処理を止めてタスクボードを「ブロック」状態に。人間の確認を待つ。

朝

CEO が朝起きると成果＋確認待ちリストが届いている

「昨晩の完了タスク一覧」と「CEOの判断が必要なブロック中タスク」だけを確認すればよい状態になっている。

# Python司令官：LLMを使わない高速・決定論的ルーティング
import os
from groq import Groq
import google.generativeai as genai

groq_client = Groq(api_key=os.environ["GROQ_API_KEY"])
genai.configure(api_key=os.environ["GEMINI_API_KEY"])

class PythonOrchestrator:
    """Python if/then による軽量オーケストレーター"""

    def route_task(self, task: dict) -> str:
        # タスクタイプで分岐（LLM不使用→高速・決定論的）
        task_type = task.get("type", "")

        if task_type == "meeting_minutes":
            return self.handle_meeting_minutes(task)
        elif task_type == "research":
            return self.handle_research(task)
        elif task_type == "report":
            return self.handle_report_with_reflection(task)
        else:
            return self.handle_general(task)

    def handle_meeting_minutes(self, task):
        # Step1: ローカルLLMでマスキング
        masked = self.local_mask(task["content"])
        # Step2: Groqで高速要約（無料API）
        summary = self.groq_summarize(masked)
        # Step3: ローカルで復元
        return self.local_unmask(summary, task["mask_map"])

    def handle_report_with_reflection(self, task, max_iterations=3):
        # リフレクションループ（最大3回）
        masked_content = self.local_mask(task["content"])
        draft = self.gemini_generate(masked_content)

        for i in range(max_iterations):
            quality_score = self.groq_quality_check(draft)
            if quality_score >= 0.85:  # 品質基準クリアで早期終了
                break
            critique = self.groq_critique(draft)
            draft = self.gemini_revise(draft, critique)

        return self.local_unmask(draft, task["mask_map"])

    def local_mask(self, text: str) -> tuple:
        # Ollama（Qwen 2.5 7B）でローカルマスキング
        # 固有名詞・金額・社名を [PERSON_1], [AMOUNT_1] 等で置換
        # mask_map: {"[PERSON_1]": "田中社長"} を返す
        pass

    def groq_summarize(self, masked_text: str) -> str:
        resp = groq_client.chat.completions.create(
            model="llama-3.3-70b-versatile",
            messages=[{"role": "user", "content": masked_text}],
            max_tokens=2000
        )
        return resp.choices[0].message.content

Tips & Best Practices

プロジェクト固有のベストプラクティス

ハイブリッド設計・機密マスキング・無料API活用での注意点。

🛡️

マスキング精度の確保

Qwen 2.5 7B（日本語が強い）をマスキング担当に推奨。人名・社名・金額だけでなく、プロジェクト固有の用語リストを事前に定義しておく。復元マップは暗号化して保存。

📂

ファイル管理

mdファイルの中間生成物は/tmp/agent_work/に格納。MCP filesystemエージェントが24時間後に自動削除。最終成果物のみ/output/へ保存。

💰

コスト管理

現在使用中のAPI（Groq等）の無料枠を1日の処理上限として設定。超過しそうな場合はローカルLLMにフォールバック。リフレクション上限3回は絶対守る。

🔄

Obsidian vs クラウドファイル

共有が必要ならクラウドファイル（Google Drive/OneDrive）推奨。Obsidianはローカル完結型だが、GitHubやiCloud経由で同期可能。MCPサーバーでObsidianとエージェントを連携できる。

⚡

Mac mini vs Windows選択

Mac mini M4 Pro 64GB が最有力候補。Apple Siliconのユニファイドメモリは70Bモデルも動作可能。RTX 4060 Ti (16GB)はVRAM不足で大型モデルが動かない場合がある。

🔌

MCP活用ポイント

執事エージェントのMCP filesystemでファイル操作を自動化。Google DriveのMCPサーバーを使えば議事録の自動取得も可能。MCPx非同期での複数サーバー同時接続を検討。

✅

現在地と方針： Groqの無料APIキーで2エージェントのリフレクションループが動作確認済み（6/19試作）。どのAPIを本格採用するかは引き続き検討中。まずはGroqで機能を磨きつつ、Gemini・Sakana Fugu等と比較しながら確定していく方針。
